# Satisfactory Calculator Graph — Design

**Date:** 2026-05-15
**Repo:** https://github.com/GibbosHomeProjects/SatisfactoryCalculatorGraph
**Game version targeted:** Satisfactory 1.1 (live build)

## 1. Purpose

A node-graph production calculator for Satisfactory 1.1. The user works **forward** from raw resource sources (miners, extractors) and builds production chains step-by-step, picking the recipe at each machine. The app computes throughput, machine counts, power consumption, somersloop effects, and belt/pipe tier warnings automatically, and supports multiple independent pipelines on the same canvas.

This is a single-user personal tool. All persistence is local to the user's browser — nothing is uploaded.

## 2. Scope

### In scope

- Forward-direction chain building. User starts from a source node and pulls connections out to add downstream machines.
- All source-node types: Miners (Mk1/2/3 × Impure/Normal/Pure), Water Extractors, Oil Pumps, Resource Well Pressurizers + satellites, Frackers.
- All production buildings: Smelter, Constructor, Assembler, Foundry, Manufacturer, Refinery, Blender, Packager, Particle Accelerator.
- Per-recipe selection at each machine node, including alternate recipes (flagged in the UI).
- Configurable clock speed per node (0.01% – 250%, matching the game).
- Configurable somersloop slot usage per applicable machine.
- AWESOME Sink endpoint nodes that compute accumulated points-per-minute and coupon-purchase rate.
- Belt-tier (Mk1–Mk6) and pipe-tier (Mk1–Mk2) warnings on edges.
- Multiple named projects stored in `localStorage`. JSON export/import for backup or sharing.
- Local-only operation. No telemetry, no cloud, no auth. Works offline once loaded.
- Deployed as a static site to GitHub Pages.

### Out of scope

- Power generator nodes (Coal/Fuel/Nuclear). Pure consumption side only.
- Reverse / target-driven solving ("I want N plates/min — figure out the chain").
- Floor-plan / building-placement layout. Splitters and mergers are implicit math, not nodes.
- Mod support. Vanilla 1.1 only.
- Multi-user collaboration. Single-user, single-browser.
- Mobile-first design. Targeting desktop browsers; mobile will work but isn't tuned for it.

## 4. Game Data Bundle

A static JSON file `src/data/satisfactory-1.1.json` is committed to the repo. It is generated by a build-time script `scripts/generate-data.ts` that parses the game's `Docs.json` (English locale). For development without owning the game install, the script can ingest pre-extracted dumps from `github.com/dmryabov/satisfactory-docs-files`.

The script uses `satisfactory-docs-parser` (npm) as the parsing library and re-emits a normalized shape (below). It is run manually by the developer when the game updates; output is committed.

### Data shape

```ts
type Form = "solid" | "fluid" | "gas";

type Item = {
  id: string;             // e.g., "iron-ore"
  displayName: string;
  iconUrl: string;        // relative path under /assets/items/
  form: Form;
  sinkPoints: number;     // 0 if cannot be sunk
  stackSize: number;      // informational
};

type BuildingCategory =
  | "miner" | "water-extractor" | "oil-pump"
  | "resource-well-pressurizer" | "resource-well-extractor"
  | "smelter" | "constructor" | "assembler" | "foundry"
  | "manufacturer" | "refinery" | "blender" | "packager"
  | "particle-accelerator" | "awesome-sink";

type Building = {
  id: string;
  displayName: string;
  category: BuildingCategory;
  basePowerMW: number;         // for Particle Accelerator: nominal mid-range
  powerIsVariable: boolean;    // true for Particle Accelerator
  somersloopSlots: number;     // 0 if not amplifiable (Packager, extractors, sink)
};

type Recipe = {
  id: string;
  displayName: string;
  buildingId: string;
  isAlternate: boolean;
  durationSeconds: number;
  inputs:  { itemId: string; amountPerCycle: number }[];
  outputs: { itemId: string; amountPerCycle: number }[];
  // For Particle Accelerator recipes only:
  variablePowerRange?: { minMW: number; maxMW: number };
};

type SourceTables = {
  minerOutputPerMin: {
    [mk in "mk1"|"mk2"|"mk3"]: { impure: number; normal: number; pure: number }
  };
  beltTierPerMin:  { mk1:60; mk2:120; mk3:270; mk4:480; mk5:780; mk6:1200 };
  pipeTierPerMin:  { mk1:300; mk2:600 };
  waterExtractorPerMin: 120;
  oilPumpPerMin:        { impure: 60; normal: 120; pure: 240 };
  // Resource well satellites: per-satellite output varies by node purity AND
  // by fluid type (Nitrogen vs Crude Oil vs Water). Exact rates are populated
  // from Docs.json by the generator; the shape here is illustrative only.
  resourceWellSatellitePerMin: Record<string /*itemId*/, { impure: number; normal: number; pure: number }>;
};
```

### Parsing notes (must be respected by the generator script)

- Fluid amounts in `Docs.json` are encoded in **millilitres**. Divide by 1000 to get cubic metres.
- Recipe `mManufactoringDuration` is seconds.
- `items_per_min = (60 / durationSeconds) × amountPerCycle`.
- Particle Accelerator recipes carry a `mVariablePowerConsumptionConstant` and `mVariablePowerConsumptionFactor`; record the min/max for display only — the calculator uses the midpoint for total-power math and flags the variability in the UI.

## 5. Graph Model

### Node types

| Node | Purpose | Inputs to model | Outputs |
| --- | --- | --- | --- |
| `MinerNode` | Ore source | `itemId`, `mk` ∈ {1,2,3}, `purity` ∈ {impure, normal, pure}, `clockPct` | one item edge |
| `WaterExtractorNode` | Water source | `clockPct` | one fluid edge (water) |
| `OilPumpNode` | Crude oil source | `purity`, `clockPct` | one fluid edge (crude oil) |
| `FrackerNode` | Pressurised fluid source | `wellItemId`, `satelliteCount`, per-satellite `purity[]`, `clockPct` | one fluid edge |
| `MachineNode` | Production step | `recipeId`, `clockPct`, `somersloopsUsed` | edges per recipe output |
| `AwesomeSinkNode` | Endpoint sink | accepts any item | reports points/min and coupons/min |

Each node has typed input and output handles (solid vs fluid) so React Flow can reject mismatched connections at connect time.

### Edges

Each edge carries:

```ts
type EdgeFlow = {
  itemId: string;
  amountPerMin: number;   // computed
  beltOrPipeTier: "mk1" | ... | "mk6";  // user-chosen per edge
};
```

Edges are visually styled by item form (solid = blue, fluid = orange, gas = green) and labelled with item name + throughput.

### Implicit splitters and mergers

- A single output handle can connect to multiple downstream consumers. Throughput is divided across outgoing edges in proportion to downstream **demand**. If total downstream demand exceeds supply, each consumer is scaled down proportionally and an under-supply warning surfaces on the affected node.
- Multiple incoming edges with the same `itemId` on a machine's input handle are summed automatically (implicit merger).
- This is deliberate: the graph models the production logic, not the floor-plan layout.

### Validation

- Type-check at connect time: solid handles only connect to solid handles; same for fluids.
- After connect: warn (do not block) if no recipe consumes the supplied item, or if downstream demand and upstream supply differ.
- Per edge: warn if `amountPerMin` exceeds the selected belt/pipe tier capacity.

## 6. Calculation Engine

A pure-function module `src/engine/compute.ts`. Inputs: the full graph state (nodes + edges + data tables). Output: a derived map `nodeId → ComputedNodeState` and `edgeId → ComputedEdgeFlow`.

### Per-node formulas

**Source nodes**

- Miner: `output = base[mk][purity] × clockPct / 100`
- Water Extractor: `output = 120 × clockPct / 100`
- Oil Pump: `output = oilPumpPerMin[purity] × clockPct / 100`
- Fracker (pressuriser drives all attached satellites): each satellite emits `resourceWellSatellitePerMin[purity_of_that_satellite] × clockPct / 100`. Pressuriser node sums them and emits a single edge.

**Production nodes**

Given recipe with `durationSeconds`, `inputs[]`, `outputs[]`:

- `nominal_output_per_min_per_machine[i] = (60 / durationSeconds) × outputs[i].amountPerCycle × clockPct/100 × output_multiplier`
- `nominal_input_per_min_per_machine[j]  = (60 / durationSeconds) × inputs[j].amountPerCycle  × clockPct/100`
- `output_multiplier = 1 + (somersloopsUsed / building.somersloopSlots)` (1 if slots = 0)
- `power_multiplier = output_multiplier ^ 2`

**Machine count.** A machine consumes all inputs at the rates above per machine. Given incoming supply `supply[j]` for each input item, the maximum machines this supply can feed is `floor_machines = min_j ( supply[j] / nominal_input_per_min_per_machine[j] )`. The engine uses `machine_count_fractional = floor_machines` and reports:
- `exact = floor_machines` (e.g., 3.5)
- `integer_ceil = ceil(floor_machines)` (e.g., 4 — what the user would actually build, implying the last machine runs at a lower effective clock)

The total per-output throughput emitted by the node is `nominal_output_per_min_per_machine[i] × machine_count_fractional`.

- `power_per_machine_MW = basePowerMW × (clockPct/100) ^ 1.321928 × power_multiplier`
- `total_power_MW = power_per_machine_MW × machine_count_fractional` (the inspector also shows the power figure assuming `integer_ceil` machines with the last one underclocked)

The exponent `1.321928` is `log₂(2.5)`. This is the documented Satisfactory overclock power scaling and applies to all production buildings and miners; it does **not** apply to generators (out of scope).

**Particle Accelerator** flags `powerIsVariable = true`. The engine uses the midpoint of `variablePowerRange` for totals and the inspector panel shows the min/max range explicitly.

**AWESOME Sink**

- `points_per_min = Σ ( edge.amountPerMin × item.sinkPoints )` summed over all incoming edges where `item.sinkPoints > 0`.
- `coupon_cost(n) = 250 × (ceil(n / 3) − 1)² + 1000` for n ≤ 2998, otherwise `249,501,250`. `n` is the running total of coupons already purchased in the save; the calculator displays "next coupon costs X points → Y minutes at current rate" using a user-editable `couponsAlreadyPurchased` field on the node (default 0).
- Throughput cap: a single sink can process up to 1200 items/min (Mk6 belt). Warn if any single incoming edge exceeds this.

### Edge propagation

The graph is a DAG in the common case; cycles do appear (e.g., the Heavy Oil Residue → Polymer Resin / Plastic / Rubber loop). The engine:

1. Computes source-node outputs (no inputs needed).
2. Walks the graph in Kahn topological order for the acyclic portion.
3. For any remaining strongly-connected component (cycle), runs a fixed-point iteration: re-evaluate every node in the SCC, clamping rates by available supply, until rates change less than 0.001/min between iterations or 50 iterations are reached. Diverging cycles surface a warning on the affected nodes; the engine does not throw.

### Project-level summary

A separate pure function `summarise(computed) → ProjectSummary`:

- Total raw inputs (sum of all source outputs grouped by item)
- Total power draw (sum across all non-source production nodes)
- Total points/min across all AWESOME Sinks
- List of warnings: under-supply, over-supply, belt/pipe tier exceeded, single-sink overload, cycle did not converge

## 7. UX

### Layout

```
┌──────────────────────────────────────────────────────────────────┐
│ Top: [Project ▼] [New] [Rename] [Duplicate] [Export] [Import]     │
├──────────────┬────────────────────────────────────┬──────────────┤
│ Node palette │      React Flow canvas             │ Inspector    │
│ (drag-out)   │      pan / zoom / minimap          │ for selected │
│              │                                    │ node or edge │
├──────────────┴────────────────────────────────────┴──────────────┤
│ Summary bar: raw inputs · total power · points/min · warnings    │
└──────────────────────────────────────────────────────────────────┘
```

### Core interactions

1. **Add a source.** Drag a source-type from the palette onto the canvas. Inspector opens; pick item (for miners), Mk, purity, clock %. Node label shows live output rate.
2. **Add a machine.** Drag a `Machine` node onto the canvas. Drag a connection from a source's output handle to the machine's input handle.
3. **Pick a recipe.** The recipe dropdown on a machine is filtered to recipes that consume at least one item available on incoming edges. Alternates are sorted after defaults and tagged "ALT". The full unfiltered list is reachable via a "show all" toggle for the rare case the user wants to add a recipe before connecting inputs.
4. **Tune.** Clock %, somersloop slot usage (slider 0..max), belt/pipe tier per edge. Everything downstream recomputes immediately.
5. **Add a sink.** Drag an `AWESOME Sink`, connect any item edge. Inspector shows points/min, coupon cost for next coupon, and minutes-until-next-coupon.

### Visual feedback

- Edge colour: solid = blue, fluid = orange, gas = green.
- Edge label: `<item-name> · <X>/min`.
- Edge red + warning icon when throughput > selected tier.
- Node yellow border + tooltip when under- or over-supplied, or when a Particle Accelerator's variable power range matters.
- Cycle non-convergence: red dashed outline on affected nodes.

### Persistence

- Current project auto-saves to `localStorage` on every change, debounced 300 ms.
- `localStorage` keys: `sat-calc:projects:index` (array of project metadata), `sat-calc:projects:<uuid>` (per-project graph blob).
- Project picker dropdown lists all saved projects. "New" creates blank, "Rename" / "Duplicate" / "Delete" act on the current one.
- **Export** triggers a download of `<project-name>.satgraph.json` containing the full project blob plus a schema version.
- **Import** accepts the same file; if schema version is older, run a migration function; if newer, refuse and prompt the user to update the app.
- Nothing is uploaded; nothing leaves the browser. No fetch calls at runtime beyond the initial static-asset load.

## 9. Testing

- **Engine** has unit tests for every formula in §6 (Vitest). Test cases cover: nominal flow, clocked >100% (verifies the `^1.321928` curve), under-supply, over-supply, somersloop multipliers per slot-class building, Particle Accelerator variable-power flag, cycle convergence on the Plastic/Rubber loop.
- **Storage** has tests for the schema migration path and the export/import round-trip.
- **UI** smoke test (Playwright) for one happy path: drag a Miner, connect a Smelter, connect a Constructor, see the output rate, save the project, reload the page, see it restored.

## 10. Deployment

- `vite.config.ts` sets `base: "/SatisfactoryCalculatorGraph/"` so asset URLs are correct under the GitHub Pages subpath.
- `.github/workflows/deploy.yml` runs `npm ci && npm run build` and publishes `dist/` to the `gh-pages` branch via `actions/deploy-pages`.
- No secrets or environment variables are required for the deploy.

## 11. Risks / Open Questions

- **Icon assets.** Item icons are not in `Docs.json`. We will extract them from the game install or pull from `github.com/dmryabov/satisfactory-docs-files`. The generator script must produce a consistent slug for each icon filename.
- **Resource Well satellite purity input.** A pressuriser can drive satellites of differing purities. The UI needs a per-satellite purity picker (a small editable list) rather than a single dropdown.
- **Cycle convergence.** The 50-iteration fixed-point solver is conservative; in practice most real cycles converge in <10. If a real-world chain fails to converge, the warning is sufficient — the user can break the cycle manually.
- **Mobile.** Out of scope. We will not break on mobile (React Flow handles touch), but no specific tuning.
